# Запуск, Деплой І Збір Відгуків

Цей гайд описує практичний спосіб запустити проєкт локально, викласти його безкоштовно і збирати якісний фідбек без зайвої інфраструктури.

## 1. Як запустити проєкт локально

### Вимоги

- `Node.js 20+` або актуальний LTS
- `npm`

### Встановлення залежностей

У корені проєкту виконайте:

```bash
npm install
```

### Запуск у dev-режимі

```bash
npm run dev
```

Після цього відкрийте:

```text
http://localhost:3000
```

### Продакшен-перевірка локально

```bash
npm run build
npm run start
```

## 2. Важливе технічне обмеження перед деплоєм

Зараз форма лідів пише дані у файл:

- `data/leads.ndjson`

Це підходить для локального запуску, але не підходить для більшості безкоштовних serverless-хостингів, зокрема для Vercel.

Причина:

- на Vercel функції мають `read-only filesystem`
- запис дозволений тільки у тимчасовий `/tmp`
- тимчасовий запис не є надійним сховищем для лідів

Практичний висновок:

- локально можна лишати поточний `app/api/leads/route.ts`
- для публічного лендінгу краще не зберігати фідбек у файлову систему
- для першої валідації краще збирати відповіді через зовнішню форму

## 3. Рекомендований безкоштовний деплой

### Рекомендація

Для цього проєкту найпростіший варіант:

- фронтенд: `Vercel`
- збір фідбеку: `Google Forms` або `Tally`

Це найменш ризиковий шлях для landing-first validation.

## 4. Як викласти на Vercel безкоштовно

### Варіант через GitHub

1. Створіть репозиторій на GitHub.
2. Завантажте код у репозиторій.
3. Зайдіть у Vercel.
4. Натисніть `Add New Project`.
5. Імпортуйте GitHub-репозиторій.
6. Vercel сам розпізнає Next.js.
7. Натисніть `Deploy`.

### Команди для Git

```bash
git init
git add .
git commit -m "Initial landing"
git branch -M main
git remote add origin <YOUR_GITHUB_REPO_URL>
git push -u origin main
```

### Що Vercel виставить автоматично

- HTTPS
- preview deployments
- production URL
- автоматичний redeploy після `git push`

## 5. Що саме треба змінити перед деплоєм

Щоб не втрачати ліди на проді, рекомендований варіант такий:

### Варіант A. Найшвидший

Замінити submit форми на зовнішню форму:

- `Google Forms` якщо потрібен найпростіший збір у таблицю
- `Tally` якщо потрібен акуратніший UX і вбудовування в лендінг

У такому сценарії:

- Vercel хостить лендінг
- форма веде на зовнішній сервіс
- всі відповіді збираються поза застосунком

### Варіант B. Залишити вбудовану форму

Потрібно замінити файлове збереження на зовнішнє сховище:

- Google Sheets через webhook/automation
- Airtable
- Supabase
- Formspree

Для етапу валідації це вже складніше, ніж потрібно.

## 6. Найпростіший спосіб збирати відгуки

### Рекомендований формат

Не намагайтесь одразу збирати "все". Для цього лендінгу краще збирати 2 типи сигналів:

1. `structured lead intake`
2. `qualitative feedback`

### 6.1 Structured lead intake

Це основна форма на сторінці. Вона має відповідати на питання:

- у людини справді є `design partners / paid pilots / early paid users`?
- скільки активних акаунтів вони ведуть?
- що в них губиться найчастіше?
- чи готові вони до `discovery call`?
- чи готові вони до `paid validation pilot`?

### 6.2 Qualitative feedback

Окремо варто збирати короткий відкритий фідбек після перегляду лендингу.

Мінімальний набір питань:

1. `Що в цьому лендінгу було найбільш зрозумілим?`
2. `Що звучить розмито або не викликає довіри?`
3. `Чи це проблема, яку ви справді маєте зараз?`
4. `Як ви вирішуєте це сьогодні?`
5. `Чи погодились би ви на 20-30 хв discovery call?`

## 7. Практичний сценарій збору фідбеку

### Найкращий ранній процес

1. Запустити лендінг на Vercel.
2. Основну CTA-форму вести у `Google Forms` або `Tally`.
3. Кожен новий лід зберігати в `Google Sheets`.
4. Раз на день вручну переглядати відповіді.
5. Відмічати окремо:
   - релевантні команди
   - нерелевантні команди
   - готових на discovery call
   - готових на paid pilot
6. Писати follow-up лише релевантним лідам.

## 8. Якщо обрати Google Forms

Це найшвидший варіант.

Що робити:

1. Створити форму в Google Forms.
2. Додати ті самі поля, які зараз є в лендінгу.
3. Увімкнути збереження відповідей у Google Sheets.
4. Поставити посилання на форму як основний CTA або відкрити її в новій вкладці.

Плюси:

- безкоштовно
- відповіді одразу в таблиці
- легко фільтрувати та сортувати
- легко вручну проставляти статуси

Мінуси:

- слабший UX
- форма виглядає окремо від сайту

## 9. Якщо обрати Tally

Це кращий варіант, якщо важливий вигляд.

Що робити:

1. Створити форму в Tally.
2. Перенести туди всі кваліфікаційні поля.
3. Вбудувати форму в лендінг або відкрити її по кнопці.
4. Налаштувати збереження відповідей у таблицю або інший інтегрований інструмент.

Плюси:

- виглядає краще за Google Forms
- легко вбудовується
- підходить для landing validation

Мінуси:

- залежність від зовнішнього сервісу
- частина кастомізації може впиратись у тариф

## 10. Які відгуки збирати в першу чергу

Не всі відгуки однаково корисні.

Пріоритетні сигнали:

- "у нас саме це зараз ламається"
- "ми ведемо це в таблицях/нотатках/Slack"
- "так, готові на дзвінок"
- "так, готові обговорювати paid pilot"

Слабкі сигнали:

- "цікаво"
- "може комусь підійде"
- "класний дизайн"
- "я б спробував колись"

## 11. Як вести таблицю з відгуками

Рекомендовані колонки в `Google Sheets`:

- `Created at`
- `Name`
- `Email`
- `Company`
- `Stage`
- `Active accounts`
- `Biggest gap`
- `Discovery call readiness`
- `Paid pilot readiness`
- `ICP fit`
- `Pain level`
- `Follow-up status`
- `Notes`

### Простий manual scoring

- `ICP fit`: `high / medium / low`
- `Pain level`: `high / medium / low`
- `Follow-up status`: `new / contacted / call booked / not relevant`

## 12. Що рекомендую саме для цього проєкту

На поточному етапі найкращий варіант такий:

1. Задеплоїти сайт на `Vercel`.
2. Не використовувати файлове збереження лідів у проді.
3. Підключити `Google Forms` або `Tally` як форму збору заявок.
4. Зберігати відповіді в `Google Sheets`.
5. Валідувати не трафік сам по собі, а якість заявок і готовність до розмови.
